# Claude Code 插件系统团队成员介绍

## 团队概述

我们的团队由5位专业成员组成，每位成员负责维护和推广一个核心插件系统。以下是各成员及其负责的插件详细介绍。

---

## 团队成员一：架构工程师

### 负责人
- **成员**：架构工程师
- **角色**：架构工程师
- **专长**：代码库分析、架构设计、功能开发流程

### 插件名称
**Feature Dev（功能开发插件）**

### 插件是干嘛的

这是一个全面、结构化的功能开发工作流程，配备专门的代理用于代码库探索、架构设计和质量审查。

它通过7个阶段引导用户完成功能开发：
1. **发现阶段**：了解需要构建什么，询问问题和约束
2. **代码库探索阶段**：启动多个 code-explorer 代理分析现有代码和模式
3. **澄清问题阶段**：填补需求空白，解决所有歧义
4. **架构设计阶段**：启动 code-architect 代理设计多种实施方案
5. **实施阶段**：按照选择的架构构建功能
6. **质量审查阶段**：启动 code-reviewer 代理确保代码质量
7. **总结阶段**：记录完成的内容和关键决策

核心组件包括：
- **code-explorer 代理**：通过跟踪执行路径深入分析现有代码库功能
- **code-architect 代理**：设计功能架构和实施蓝图
- **code-reviewer 代理**：审查代码中的错误、质量问题和项目约定

### 使用场景

1. **复杂功能开发**：当需要开发涉及多个文件的新功能时
   - 例如：添加用户认证系统、实现缓存层

2. **架构决策场景**：需要做出重要架构选择时
   - 例如：选择数据库方案、设计服务架构

3. **代码集成**：与现有代码进行复杂集成时
   - 例如：将新的支付系统集成到现有电商平台

4. **需求不明确**：功能需求有些模糊，需要澄清时
   - 例如："添加搜索功能"需要明确搜索范围和性能要求

### 解决的问题

1. **盲目开发问题**：避免在不了解代码库的情况下直接修改代码
   - 通过代码库探索阶段确保理解现有模式

2. **架构混乱问题**：通过设计阶段确保架构决策经过深思熟虑
   - code-architect 提供多种方案并分析权衡

3. **质量问题**：通过专门的审查代理确保代码质量
   - code-reviewer 检测 bug、约定违规和代码质量问题

4. **需求模糊问题**：通过澄清阶段确保所有需求明确
   - 在设计开始前填补所有空白和歧义

5. **开发效率低**：结构化流程减少反复试错
   - 7个阶段确保每个步骤都有明确目标

## 团队成员二：质量保障团队

### 负责人
- **成员**：质量保障团队
- **角色**：质量保障工程师
- **专长**：代码审查、质量分析、误报过滤

### 插件名称
**Code Review（代码审查插件）**

### 插件是干嘛的

使用多个专用代理和基于置信度的评分进行自动拉取请求代码审查，以过滤误报。

该插件并行启动4个代理从不同角度独立审查更改：
1. **代理 #1 & #2**：审查 CLAUDE.md 合规性，检查项目规范遵循情况
2. **代理 #3**：扫描更改中的明显 bug，专注于新增的问题
3. **代理 #4**：分析 git blame/history 以查找基于上下文的问题

置信度评分系统：
- 每个问题独立评分 0-100
- 过滤掉低于80置信度阈值的问题
- 确保只发布高质量、可操作的反馈

### 使用场景

1. **PR 审查**：对拉取请求进行自动代码审查
   - 开发者在创建 PR 前运行本地审查
   - 或在 CI/CD 流程中自动触发

2. **CI/CD 集成**：作为持续集成流程的一部分
   - 在 PR 创建或更新时自动触发
   - 使用 --comment 标志将审查作为 PR 注释发布

3. **质量门控**：在合并前确保代码质量
   - 阻止低质量代码进入主分支
   - 捕获关键问题和违规

4. **团队规范检查**：检查代码是否符合团队规范
   - 基于 CLAUDE.md 进行合规性检查
   - 确保代码风格和约定统一

### 解决的问题

1. **审查耗时问题**：通过并行代理提高审查效率
   - 4个独立代理同时工作，全面覆盖

2. **误报过多问题**：通过置信度评分系统过滤低质量反馈
   - 阈值80确保只有高置信度问题被报告
   - 减少噪音和不相关的评论

3. **审查不一致问题**：多个代理确保不同角度的审查
   - 规范检查、bug 检测、历史分析并行进行

4. **规范执行困难**：自动检查 CLAUDE.md 合规性
   - 2个规范代理确保规范检查的可靠性

5. **上下文缺失**：git blame 分析提供历史上下文
   - 理解代码变更的历史和原因

## 团队成员三：DevOps 工程师

### 负责人
- **成员**：DevOps 工程师
- **角色**：DevOps 工程师
- **专长**：Git 工作流、自动化、版本控制

### 插件名称
**Commit Commands（提交命令插件）**

### 插件是干嘛的

使用简单的命令来提交、推送和创建拉取请求，简化您的 git 工作流。该插件自动化常见的 git 操作，减少上下文切换和手动命令执行。

核心命令：
- **/commit**：自动生成提交消息并创建提交
  - 分析当前 git 状态
  - 审查暂存和未暂存的更改
  - 检查最近的提交消息以匹配仓库样式
  - 起草适当的提交消息并创建提交

- **/commit-push-pr**：一步完成提交、推送和创建 PR
  - 创建新分支（如果当前在 main 上）
  - 提交更改
  - 推送到 origin
  - 使用 gh pr create 创建拉取请求
  - 提供包含摘要和测试计划的 PR 描述

- **/clean_gone**：清理已从远程存储库删除的本地分支
  - 列出所有本地分支以识别 [gone] 状态
  - 删除与 [gone] 分支关联的 worktrees
  - 删除所有标记为 [gone] 的分支

### 使用场景

1. **日常提交**：开发过程中快速提交更改
   - 完成一小部分功能后运行 /commit
   - 自动生成符合仓库风格的提交消息

2. **功能分支工作流**：将功能开发、提交、创建 PR 整合为一条命令
   - 开发完成后直接运行 /commit-push-pr
   - 自动处理分支创建、提交、推送、PR 创建

3. **分支维护**：定期清理已合并的分支
   - PR 合并后运行 /clean_gone
   - 保持本地分支列表整洁

4. **减少上下文切换**：避免在多个 git 命令之间来回切换
   - 一条命令完成多个 git 操作
   - 减少开发者认知负担

### 解决的问题

1. **提交消息编写困难**：自动分析更改并生成符合仓库风格的提交消息
   - 遵循传统提交惯例
   - 从现有提交历史学习仓库样式
   - 自动包含 Claude Code 归属

2. **工作流程繁琐**：将多个 git 命令整合为单一命令
   - 避免运行 git add、git commit、git push、gh pr create 等多个命令
   - 减少上下文切换和记忆负担

3. **分支管理混乱**：自动清理过时的本地分支
   - 自动识别 [gone] 分支
   - 安全清理关联的 worktrees
   - 保持本地环境整洁

4. **效率低下**：减少手动命令执行，提高开发效率
   - 自动化常见的 git 操作
   - 让开发者专注于代码而非命令

5. **PR 创建复杂**：自动生成包含测试计划的 PR 描述
   - 分析分支中的所有提交生成全面描述
   - 包含测试计划检查表

## 团队成员四：插件开发专家

### 负责人
- **成员**：插件开发专家
- **角色**：插件开发专家
- **专长**：插件架构、MCP 集成、钩子系统

### 插件名称
**Plugin Dev（插件开发工具包）**

### 插件是干嘛的

一个用于开发 Claude Code 插件的综合工具包，提供有关钩子、MCP 集成、插件结构和市场发布的专家指导。包含7个专门技能帮助用户构建高质量插件。

核心技能：
1. **钩子开发技能**：高级钩子 API 和事件驱动的自动化
   - 基于提示的钩子（推荐），具有 LLM 决策能力
   - 用于确定性验证的命令钩子
   - 所有钩子事件：PreToolUse、PostToolUse、Stop、SubagentStop、SessionStart、SessionEnd、UserPromptSubmit、PreCompact、Notification
   - 包含3个示例钩子脚本和实用程序

2. **MCP 集成技能**：Model Context Protocol 服务器集成
   - MCP 服务器配置（.mcp.json vs plugin.json）
   - 所有服务器类型：stdio（本地）、SSE（托管/OAuth）、HTTP（REST）、WebSocket（实时）
   - 环境变量扩展和身份验证模式

3. **插件结构技能**：插件组织和清单配置
   - 标准插件目录结构和自动发现
   - plugin.json 清单格式和所有字段
   - 组件组织和命名约定

4. **插件设置技能**：使用 .local.md 文件的配置模式
   - YAML frontmatter + markdown body 结构
   - bash 脚本的解析技术
   - 临时活动的钩子和状态管理

5. **命令开发技能**：创建斜杠命令
   - 斜杠命令结构和 markdown 格式
   - YAML frontmatter 字段和动态参数
   - 命令组织和命名空间

6. **代理开发技能**：创建自主代理
   - 代理文件结构和所有 frontmatter 字段
   - 描述格式用于可靠触发
   - 系统提示设计模式

7. **技能开发技能**：创建具有强触发器的技能
   - 技能结构和渐进式披露原则
   - 有力触发器描述和写作风格

8. **/plugin-dev:create-plugin 命令**：端到端插件创建工作流
   - 8个阶段流程：发现、组件规划、详细设计、结构创建、组件实现、验证、测试、文档
   - 类似 feature-dev 的结构化流程

### 使用场景

1. **创建新插件**：从零开始开发 Claude Code 插件
   - 从概念到完整的插件项目
   - 自动创建目录结构、配置文件、实现代码

2. **集成外部服务**：通过 MCP 连接外部 API 和服务
   - 集成数据库、API、工具等外部服务
   - 配置不同类型的 MCP 服务器

3. **添加自动化**：通过钩子系统实现事件驱动的自动化
   - 工具使用前验证
   - 会话开始时注入上下文
   - 会话结束时清理资源

4. **插件配置管理**：实现插件设置和状态管理
   - 使用 .local.md 文件存储项目级配置
   - 实现用户偏好和插件状态

### 解决的问题

1. **插件开发门槛高**：提供完整的开发指南和工作流
   - 7个技能覆盖插件开发的所有方面
   - 渐进式披露减少上下文负担
   - 提供丰富的示例和实用程序

2. **集成复杂**：简化 MCP 服务器和外部服务集成
   - 提供所有服务器类型的配置示例
   - 详细说明身份验证和环境变量使用
   - 集成模式和性能优化指南

3. **自动化困难**：通过钩子系统轻松实现自动化
   - 基于提示和命令钩子的完整文档
   - 事件触发和输出格式说明
   - 安全最佳实践和输入验证

4. **文档不完整**：提供渐进式披露的完整文档体系
   - 元数据（始终加载）、核心文档（触发时加载）、详细参考（按需加载）
   - 约 11,065 字的核心文档 + 10,000+ 字的详细指南
   - 12+ 个工作示例和 6 个实用程序

5. **缺乏最佳实践**：强调安全和可移植性
   - 安全优先的方法和验证实用程序
   - 全程使用 ${CLAUDE_PLUGIN_ROOT} 确保可移植性
   - 基于官方 Claude Code 文档

## 团队成员五：SDK 开发工程师

### 负责人
- **成员**：SDK 开发工程师
- **角色**：SDK 开发工程师
- **专长**：Agent SDK、Python/TypeScript 开发、最佳实践验证

### 插件名称
**Agent SDK Dev（Agent SDK 开发插件）**

### 插件是干嘛的

用于创建和验证 Python 和 TypeScript Claude Agent SDK 应用程序的全面插件。它简化了构建 Agent SDK 应用程序的整个生命周期，从初始脚手架到根据最佳实践进行验证。

核心组件：
- **/new-sdk-app 命令**：交互式创建新的 Agent SDK 应用程序
  - 询问关于项目的澄清问题（语言、名称、代理类型、起点）
  - 检查并安装最新的 SDK 版本
  - 创建所有必要的项目文件和配置
  - 设置适当的环境文件（.env.example、.gitignore）
  - 提供适合用例的工作示例
  - 运行类型检查或语法验证
  - 使用适当的验证器代理自动验证设置

- **agent-sdk-verifier-py 代理**：彻底验证 Python Agent SDK 应用程序
  - SDK 安装和版本检查
  - Python 环境设置验证
  - 正确的 SDK 使用和模式检查
  - 代理初始化和配置验证
  - 环境和安全验证（.env、API 密钥）
  - 错误处理和功能检查
  - 文档完整性验证

- **agent-sdk-verifier-ts 代理**：彻底验证 TypeScript Agent SDK 应用程序
  - SDK 安装和版本检查
  - TypeScript 配置验证（tsconfig.json）
  - 正确的 SDK 使用和模式检查
  - 类型安全和导入验证
  - 代理初始化和配置验证
  - 环境和安全验证
  - 错误处理和功能检查
  - 文档完整性验证

### 使用场景

1. **新项目创建**：快速启动新的 Agent SDK 项目
   - 选择语言（TypeScript 或 Python）
   - 选择代理类型（编码、业务、自定义）
   - 选择起点（最小、基本或特定示例）
   - 自动设置完整项目结构

2. **项目验证**：验证现有项目是否符合最佳实践
   - 创建新项目后自动运行验证器
   - 修改现有项目后手动触发验证
   - 部署前确保遵循最佳实践

3. **部署前检查**：在部署到生产环境前验证设置
   - 检查 SDK 版本是否最新
   - 验证配置文件和环境设置
   - 确保没有阻止部署的问题

4. **学习 SDK 用法**：了解官方文档模式
   - 提供符合官方文档的工作示例
   - 验证器指出与最佳实践的偏差
   - 带有 SDK 文档引用的具体建议

### 解决的问题

1. **项目初始化复杂**：自动检查并安装最新 SDK 版本
   - 避免手动检查和安装
   - 确保项目使用最新功能

2. **最佳实践难以遵循**：通过验证器确保代码符合官方模式
   - 自动检查所有最佳实践
   - 提供带有文档引用的具体建议
   - 防止常见错误和反模式

3. **环境配置繁琐**：自动创建必要的配置文件
   - 创建 .env.example 和 .gitignore
   - 设置正确的环境结构
   - 提醒用户设置 API 密钥

4. **类型检查困难**：自动运行类型检查和语法验证
   - TypeScript 项目自动运行 tsc --noEmit
   - Python 项目验证语法
   - 在早期发现问题

5. **文档不完整**：验证文档完整性
   - 检查 README 是否存在
   - 验证环境配置说明
   - 确保 API 密钥使用说明清晰

## 团队协作流程

### 推荐的工作流程

```
1. 架构阶段（架构工程师）
   ↓ /feature-dev
   探索代码库、设计架构

2. 开发阶段（各成员协作）
   ↓ /commit (DevOps 工程师)
   日常开发提交

3. 质量检查阶段（质量保障团队）
   ↓ /code-review
   审查代码质量

4. 部署准备阶段（DevOps 工程师）
   ↓ /commit-push-pr
   创建 PR

5. 后续开发（SDK 开发工程师）
   ↓ /new-sdk-app
   创建新的 SDK 应用
```

### 插件生态系统

我们的插件系统相互配合，形成完整的开发生态：

| 插件 | 主要功能 | 与其他插件的协作 |
|--------|---------|-------------------|
| Feature Dev | 架构设计和开发指导 | → 为其他插件提供架构基础 |
| Code Review | 代码质量审查 | → 确保 Commit Commands 的代码质量 |
| Commit Commands | Git 工作流简化 | → 简化所有插件的提交流程 |
| Plugin Dev | 插件能力扩展 | → 扩展整个插件生态系统 |
| Agent SDK Dev | SDK 应用支持 | → 为新插件提供 SDK 项目模板 |

## 快速开始指南

### 新用户推荐路径

1. **第一步**：安装 Commit Commands 插件
