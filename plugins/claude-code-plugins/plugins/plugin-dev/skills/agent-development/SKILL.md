# Claude Code 插件的代理开发

## 概述

代理是独立处理复杂、多步骤任务的自主子进程。理解代理结构、触发条件和系统提示设计能够创建强大的自主能力。

**核心概念：**
- 代理用于自主工作，命令用于用户发起的操作
- 带有 YAML 前置元数据的 Markdown 文件格式
- 通过描述字段中的示例进行触发
- 系统提示定义代理行为
- 模型和颜色自定义

## 代理文件结构

### 完整格式

```markdown
---
name: agent-identifier
description: 当[触发条件]时使用此代理。示例：

<example>
Context: [情况描述]
user: "[用户请求]"
assistant: "[助手应如何响应并使用此代理]"
<commentary>
[为什么应该触发此代理]
</commentary>
</example>

<example>
[附加示例...]
</example>

model: inherit
color: blue
tools: ["Read", "Write", "Grep"]
---

你是[代理角色描述]...

**你的核心职责：**
1. [职责 1]
2. [职责 2]

**分析过程：**
[逐步工作流]

**输出格式：**
[返回什么]
```

## 前置元数据字段

### name（必需）

用于命名空间和调用的代理标识符。

**格式：** 小写、数字、仅限连字符
**长度：** 3-50 个字符
**模式：** 必须以字母数字开头和结尾

**好的示例：**
- `code-reviewer`
- `test-generator`
- `api-docs-writer`
- `security-analyzer`

**坏的示例：**
- `helper`（太通用）
- `-agent-`（以连字符开头/结尾）
- `my_agent`（不允许下划线）
- `ag`（太短，< 3 个字符）

### description（必需）

定义 Claude 何时应触发此代理。**这是最关键的字段。**

**必须包含：**
1. 触发条件（"当...时使用此代理"）
2. 多个显示用法的 `<example>` 块
3. 每个示例中的上下文、用户请求和助手响应
4. 解释为什么触发代理的 `<commentary>`

**格式：**
```
当[条件]时使用此代理。示例：

<example>
Context: [场景描述]
user: "[用户所说的]"
assistant: "[Claude 应如何响应]"
<commentary>
[为什么此代理是合适的]
</commentary>
</example>

[更多示例...]
```

**最佳实践：**
- 包含 2-4 个具体示例
- 展示主动和被动触发
- 覆盖相同意图的不同措辞
- 在注释中解释推理
- 具体说明何时不使用代理

### model（必需）

代理应使用哪个模型。

**选项：**
- `inherit` - 使用与父级相同的模型（推荐）
- `sonnet` - Claude Sonnet（平衡型）
- `opus` - Claude Opus（最强大，昂贵）
- `haiku` - Claude Haiku（快速，便宜）

**推荐：** 除非代理需要特定模型功能，否则使用 `inherit`。

### color（必需）

代理在 UI 中的视觉标识符。

**选项：** `blue`、`cyan`、`green`、`yellow`、`magenta`、`red`

**指导原则：**
- 为同一插件中的不同代理选择不同的颜色
- 为类似的代理类型使用一致的颜色
- 蓝色/青色：分析、审查
- 绿色：以成功为导向的任务
- 黄色：警告、验证
- 红色：关键、安全
- 洋红色：创造性、生成

### tools（可选）

将代理限制为特定工具。

**格式：** 工具名称数组

```yaml
tools: ["Read", "Write", "Grep", "Bash"]
```

**默认：** 如果省略，代理有权访问所有工具

**最佳实践：** 将工具限制为所需的最小值（最小权限原则）

**常用工具集：**
- 只读分析：`["Read", "Grep", "Glob"]`
- 代码生成：`["Read", "Write", "Grep"]`
- 测试：`["Read", "Bash", "Grep"]`
- 完全访问：省略字段或使用 `["*"]`

## 系统提示设计

Markdown 正文成为代理的系统提示。以第二人称编写，直接称呼代理。

### 结构

**标准模板：**
```markdown
你是[角色]，专门从事[领域]。

**你的核心职责：**
1. [主要职责]
2. [次要职责]
3. [附加职责...]

**分析过程：**
1. [第一步]
2. [第二步]
3. [第三步]
[...]

**质量标准：**
- [标准 1]
- [标准 2]

**输出格式：**
按以下格式提供结果：
- [包含什么]
- [如何结构化]

**边缘情况：**
处理这些情况：
- [边缘情况 1]：[如何处理]
- [边缘情况 2]：[如何处理]
```

### 最佳实践

✅ **应该：**
- 以第二人称编写（"你是..."、"你将..."）
- 具体说明职责
- 提供逐步过程
- 定义输出格式
- 包含质量标准
- 处理边缘情况
- 保持在 10,000 个字符以下

❌ **不应该：**
- 以第一人称编写（"我是..."、"我将..."）
- 含糊或通用
- 省略过程步骤
- 让输出格式未定义
- 省略质量指导
- 忽略错误情况

## 创建代理

### 方法 1：AI 辅助生成

使用此提示模式（从 Claude Code 提取）：

```
基于此请求创建代理配置："[您的描述]"

要求：
1. 提取核心意图和职责
2. 为领域设计专家人设
3. 创建包含以下内容的综合系统提示：
   - 清晰的行为边界
   - 具体方法论
   - 边缘情况处理
   - 输出格式
4. 创建标识符（小写、连字符、3-50 个字符）
5. 编写带有触发条件的描述
6. 包含 2-3 个显示何时使用的 <example> 块

返回 JSON：
{
  "identifier": "agent-name",
  "whenToUse": "当...时使用此代理。示例：<example>...</example>",
  "systemPrompt": "你是..."
}
```

然后转换为带有前置元数据的代理文件格式。

有关完整模板，请参阅 `examples/agent-creation-prompt.md`。

### 方法 2：手动创建

1. 选择代理标识符（3-50 个字符，小写，连字符）
2. 编写带有示例的描述
3. 选择模型（通常是 `inherit`）
4. 选择颜色用于视觉识别
5. 定义工具（如果限制访问）
6. 使用上述结构编写系统提示
7. 保存为 `agents/agent-name.md`

## 验证规则

### 标识符验证

```
✅ 有效：code-reviewer、test-gen、api-analyzer-v2
❌ 无效：ag（太短）、-start（以连字符开头）、my_agent（下划线）
```

**规则：**
- 3-50 个字符
- 仅限小写字母、数字、连字符
- 必须以字母数字开头和结尾
- 不允许下划线、空格或特殊字符

### 描述验证

**长度：** 10-5,000 个字符
**必须包含：** 触发条件和示例
**最佳：** 200-1,000 个字符，包含 2-4 个示例

### 系统提示验证

**长度：** 20-10,000 个字符
**最佳：** 500-3,000 个字符
**结构：** 清晰的职责、过程、输出格式

## 代理组织

### 插件代理目录

```
plugin-name/
└── agents/
    ├── analyzer.md
    ├── reviewer.md
    └── generator.md
```

`agents/` 中的所有 `.md` 文件都会自动发现。

### 命名空间

代理自动进行命名空间化：
- 单个插件：`agent-name`
- 带有子目录：`plugin:subdir:agent-name`

## 测试代理

### 测试触发

创建测试场景以验证代理正确触发：

1. 编写带有特定触发示例的代理
2. 在测试中使用与示例类似的措辞
3. 检查 Claude 加载代理
4. 验证代理提供预期功能

### 测试系统提示

确保系统提示完整：

1. 给代理典型任务
2. 检查它遵循过程步骤
3. 验证输出格式正确
4. 测试提示中提到的边缘情况
5. 确认满足质量标准

## 快速参考

### 最小代理

```markdown
---
name: simple-agent
description: 当...时使用此代理。示例：<example>...</example>
model: inherit
color: blue
---

你是一个执行 [X] 的代理。

过程：
1. [步骤 1]
2. [步骤 2]

输出：[提供什么]
```

### 前置元数据字段摘要

| 字段 | 必需 | 格式 | 示例 |
|-------|----------|--------|---------|
| name | 是 | lowercase-hyphens | code-reviewer |
| description | 是 | 文本 + 示示例 | 当...时使用 <example>... |
| model | 是 | inherit/sonnet/opus/haiku | inherit |
| color | 是 | 颜色名称 | blue |
| tools | 否 | 工具名称数组 | ["Read", "Grep"] |

### 最佳实践

**应该：**
- ✅ 在描述中包含 2-4 个具体示例
- ✅ 编写具体的触发条件
- ✅ 除非特定需要，否则对模型使用 `inherit`
- ✅ 选择适当的工具（最小权限）
- ✅ 编写清晰、结构化的系统提示
- ✅ 彻底测试代理触发

**不应该：**
- ❌ 使用没有示例的通用描述
- ❌ 省略触发条件
- ❌ 给所有代理相同的颜色
- ❌ 授予不必要的工具访问
- ❌ 编写含糊的系统提示
- ❌ 跳过测试

## 实现工作流程

要为插件创建代理：

1. 定义代理目的和触发条件
2. 选择创建方法（AI 辅助或手动）
3. 创建 `agents/agent-name.md` 文件
4. 编写包含所有必需字段的前置元数据
5. 遵循最佳实践编写系统提示
6. 在描述中包含 2-4 个触发示例
7. 使用 `scripts/validate-agent.sh`' 进行验证
8. 使用真实场景测试触发
9. 在插件 README 中记录代理

专注于清晰的触发条件和综合系统提示以实现自主操作。
