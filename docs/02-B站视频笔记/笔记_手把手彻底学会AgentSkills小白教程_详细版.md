## 详细内容

### [01:12] 什么是 Agent Skills？

**概念类比**：Skill（技能）就像人类的职业技能

- **厨师技能**包含：
  - 流程：先炒什么、后放什么
  - 配方：油温多高、盐放多少
  - 工具：煤气灶、锅铲
  - 独家材料：秘制辣椒酱

- **Agent Skill 同理包含**：
  - **SKILL.md** — 流程和配方（指令）
  - **References/** — 参考资料和规范
  - **Scripts/** — 可执行的脚本工具
  - **assets/** — 独家材料（图片、模板等）

### [03:04] 最简技能示例

**案例**：创建一个"秋芝创意"技能，用于餐厅物料创意生成

**SKILL.md 结构**：
```yaml
---
name: qiuzhi-creative
description: 做创意物料用的，当用户说要做个海报什么的物料，它就自动触发了
---

# 指令部分
我们的餐厅叫做秋芝餐厅
品牌的风格要求：
- 输出格式：xxx
- 配色规范：xxx
```

**创建步骤**：
1. 在 `~/.claude/skills/` 下创建文件夹
2. 文件夹名 = 技能名（如：`qiuzhi-creative`）
3. 创建必需的 `SKILL.md` 文件
4. 填写元信息（name + description）和指令

### [04:23] 按需加载机制 — 第一层：元信息层

**核心原理**：AI 只在需要时才加载完整指令

| 层级 | 加载时机 | 内容 | Token消耗 |
|------|---------|------|-----------|
| **元信息层** | 每次对话都看 | name + description | 极低 |
| **指令层** | 触发技能时 | 完整 SKILL.md | 按需 |
| **资源层** | 进一步需要时 | References/Scripts/assets | 按需 |

**优势**：
- 可以同时拥有多个技能，互不干扰
- AI 回答更精准（无其他提示词干扰）
- 大幅节省 Token 成本

### [07:32] 按需加载机制 — 第二/三层：References 和 Scripts

**问题**：当物料类型增多时，SKILL.md 会变得过长

**解决方案**：将不同类型物料的规范拆分到 References 文件夹

```
qiuzhi-creative/
├── SKILL.md              # 总指令（仅保留常见物料）
└── references/
    ├── 实体物料规格.md   # 餐盒、杯子、员工服装
    └── 社交媒体规格.md   # 公众号封面、微博配图
```

**SKILL.md 中的指引**：
```
如果要做线下物料 → 读取 references/实体物料规格.md
如果要做社交媒体图 → 读取 references/社交媒体规格.md
```

### [10:08] Scripts 文件夹 — 让技能执行任务

**功能扩展**：让 AI 直接生成图片，而不是只输出创意

**实现方式**：
1. 在 `Scripts/` 中放置可执行脚本
2. 在 SKILL.md 中告诉 AI 何时调用脚本
3. 脚本代码 AI 不需要读取（不占用 Token）

**示例脚本**：调用 Niji API 生成图片

```bash
# scripts/generate-image.sh
# 调用 Niji API 生成图片
# 参数：提示词 + logo图片路径
```

### [13:26] assets 文件夹 — 提供静态资源

**用途**：存放 logo、模板、样例等固定资源

```
qiuzhi-creative/
└── assets/
    ├── logo-horizontal.png
    └── logo-vertical.png
```

**SKILL.md 中的指引**：
```
如果生成图片，需要将 assets/ 中的 logo 图片作为参数传给脚本
```

### [14:52] Skill 完整工作流程演示

**用户输入**：帮我做一张周六面料面费的实体海报

**执行流程**：
1. AI 识别到需要使用"秋芝创意"技能
2. 检测到是"实体物料" → 加载 `references/实体物料规格.md`
3. 用户要求生成图片 → 调用 `scripts/generate-image.sh`
4. 传入 logo 图片作为参数
5. 输出符合品牌规范的图片

**结果**：尺寸、配色、logo 完全符合品牌规范

## 重要概念与术语

| 术语 | 解释 |
|------|------|
| **Agent** | 智能代理，能够自主执行任务、调用工具的 AI 系统 |
| **Skill** | 技能，为 Agent 增加特定专业能力的扩展模块 |
| **SKILL.md** | 技能的核心文件，包含元信息和指令 |
| **元信息 (Metadata)** | YAML frontmatter，包含 name 和 description |
| **指令 (Instruction)** | 具体告诉 AI 怎么做的提示词内容 |
| **References/** | 参考资料，按需加载的详细说明文档 |
| **Scripts/** | 可执行脚本，只执行不读取（不占 Token） |
| **assets/** | 静态资源文件夹 |
| **按需加载** | 只在需要时才加载内容的设计机制 |
| **Lazy Loading** | 懒加载，节省 Token 的核心技术 |

│                    第一层：元信息层                    │
│  ┌───────────────────────────────────────────────┐  │
│  │  始终加载，AI 每次对话都会看所有技能的元信息    │  │

│  │  用途：让 AI 知道自己有哪些技能可用             │  │
│  └───────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────┘
                        ↓ 触发技能后
┌─────────────────────────────────────────────────────┐
│                    第二层：指令层                     │
│  ┌───────────────────────────────────────────────┐  │
│  │  只有 AI 决定使用该技能时才加载                │  │
│  │  内容：完整的 SKILL.md 指令部分                │  │
│  │  用途：告诉 AI 具体如何执行任务                │  │
│  └───────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────┘
                        ↓ 进一步需要时
┌─────────────────────────────────────────────────────┐
│                    第三层：资源层                     │
│  ┌───────────────────────────────────────────────┐  │
│  │  按需加载，只在需要详细资料或执行脚本时才用     │  │
│  │  内容：References/Scripts/assets               │  │
│  │  用途：提供详细规范、执行脚本、静态资源         │  │
│  └───────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────┘

## 关键要点

1. **本质就是提示词** — Skill 本质上是结构化的提示词，但工程上有很大优势

2. **按需加载是核心** — 三层结构让 AI 只加载需要的内容，节省 Token

3. **Scripts 不占 Token** — 脚本只执行不读取，再多的代码也不消耗上下文

4. **不用手写代码** — 使用"秋芝技能生成器"可以零代码创建技能

5. **精选胜过堆砌** — 像游戏英雄的 4 个技能一样，只保留最高频、最需要的技能
